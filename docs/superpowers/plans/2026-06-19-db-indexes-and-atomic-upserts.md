# DB FK Indexes + Atomic Upserts Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add evidence-based FK/query indexes across the US1–US3 schema, and convert the three side-effect-free get-or-create upserts to atomic PostgreSQL `INSERT ... ON CONFLICT` so concurrent same-key inserts can't raise.

**Architecture:** Two independent changes under `src/db`. Task 1 adds 6 indexes (4 single-column via `index=True`, 2 composite via `Index()` in `__table_args__`) chosen from the actual repository filter/sort patterns, plus one Alembic migration chained after the US3 head `07c7492119e4`. Task 2 rewrites only the three repository methods whose collision behavior is "return the existing row unchanged" (`KeywordTerm` upsert_term, `DocumentChunk` get_or_add_chunk, `WaveScenarioRule` link_scenario_rule) to `insert(...).on_conflict_do_update(...).returning(...)`. `ProjectionRepository.upsert_status` and `SettingsRepository.upsert_setting` are intentionally left as select-then-write because they carry conditional side-logic (status reset / error clear, soft-delete un-delete) that does not map cleanly to a single ON CONFLICT clause.

**Tech Stack:** Python 3.12, SQLAlchemy 2.x, Alembic, psycopg 3, PostgreSQL 16, pytest.

**Issue:** #85

## Global Constraints

- Python 3.12; run all Python via `.venv/bin/python`. Branch: `feature/004-db-indexes-and-atomic-upserts` (from main, which has US1+US2+US3).
- Deterministic tests only — real local PostgreSQL via `TEST_DATABASE_URL`; skip via `@REQUIRES_DB` when unset (never silent-pass). Schema via Alembic migration (not `create_all`). Repositories flush, never commit.
- The new migration's `down_revision` MUST be `'07c7492119e4'` (the US3 head).
- **No schema semantic change.** Indexes are pure performance; the atomic upserts must preserve EXACT existing behavior (return the existing row unchanged on collision; idempotent; same return type).
- **Index only what is queried.** Do NOT index a column that no repository filters/sorts on, and do NOT add an index whose leading column already heads an existing unique constraint (that constraint's index already serves prefix lookups). `ticker_id` on notifications/portfolio_items is NOT indexed — no query filters it.
- Safety invariants unchanged: SER-001/SC-005 (no trading/order/guaranteed fields), FR-019 anonymization behavior unchanged.
- Commit after every task.

## Task 2: Atomic ON CONFLICT for side-effect-free get-or-create upserts

**Files:**
- Modify: `src/db/repositories/source_document_repository.py`
- Modify: `src/db/repositories/projection_repository.py`
- Modify: `src/db/repositories/graph_repository.py`
- Modify: `tests/test_storage_postgres_projection_status.py`

**Interfaces (unchanged signatures — behavior identical, now atomic):**
- `SourceDocumentRepository.get_or_add_chunk(source_document_id, chunk_index, chunk_text, chunk_hash, *, token_count=None, metadata=None) -> DocumentChunk`
- `ProjectionRepository.upsert_term(term, normalized_term, language=None) -> KeywordTerm`
- `GraphRepository.link_scenario_rule(scenario_id, rule_id, *, role) -> WaveScenarioRule`

Pattern: `insert(Model).values(...).on_conflict_do_update(index_elements=[...], set_={<no-op>}).returning(Model)`, then `self.session.scalars(stmt).one()`. A no-op `set_` (a column set to its own `excluded` value) is required so `RETURNING` yields the existing row on conflict (DO NOTHING would return nothing). This preserves the current "return existing unchanged, first-write-wins" semantics exactly. Conflict targets match the existing unique constraints (including `postgresql_nulls_not_distinct=True` ones, supported by PG16 ON CONFLICT inference).

- [ ] **Step 2: Rewrite `get_or_add_chunk` to atomic**

In `src/db/repositories/source_document_repository.py`, add the import at the top (with the other imports):
```python
from sqlalchemy.dialects.postgresql import insert as pg_insert
```
Replace the `get_or_add_chunk` body with:
```python
    def get_or_add_chunk(
        self,
        source_document_id,
        chunk_index: int,
        chunk_text: str,
        chunk_hash: str,
        *,
        token_count: int | None = None,
        metadata: dict | None = None,
    ) -> DocumentChunk:
        stmt = (
            pg_insert(DocumentChunk)
            .values(
                source_document_id=source_document_id,
                chunk_index=chunk_index,
                chunk_text=chunk_text,
                chunk_hash=chunk_hash,
                token_count=token_count,
                chunk_metadata=metadata if metadata is not None else {},
            )
            .on_conflict_do_update(
                index_elements=["source_document_id", "chunk_index"],
                set_={"source_document_id": DocumentChunk.source_document_id},
            )
            .returning(DocumentChunk)
        )
        return self.session.scalars(stmt).one()
```
(The `set_` is a no-op self-assignment so the existing row is returned unchanged — `chunk_text`/`chunk_hash` are NOT overwritten, matching the prior "return existing" behavior. `chunk_metadata` is the mapped attribute for the DB column `metadata`.)
