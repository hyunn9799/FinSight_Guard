# 데이터 모델: 문서 한국어화 검증 상태

## 목적

이 모델은 런타임 데이터베이스 스키마가 아니다. 번역 대상, 용어, 보호 요소, 배치와 검증 결과를 일관되게 기록하기 위한 문서 작업 모델이다.

## 개체 1: 문서 산출물

하나의 `specs/**/*.md` 파일을 나타낸다.

| 필드 | 의미 | 검증 규칙 |
|------|------|-----------|
| `path` | 저장소 기준 상대 경로 | 유일해야 하며 실제 파일을 가리켜야 함 |
| `feature_id` | 기능 번호와 짧은 이름 | 상위 디렉터리와 일치해야 함 |
| `artifact_type` | `spec`, `checklist`, `research`, `data-model`, `contract`, `plan`, `quickstart`, `tasks`, 기타 | 허용된 유형 또는 설명이 있는 `other` |
| `source_language` | 번역 시작 전 설명 문장의 주 언어 | `english`, `mixed`, `korean` 중 하나 |
| `translation_state` | 현재 번역 상태 | 정의된 상태 전이만 허용 |
| `review_state` | 검증 상태 | 모든 필수 게이트가 통과되어야 `approved` |
| `dependency_paths` | 직접 의미·링크 의존 문서 | 모든 경로가 존재하거나 누락 사유가 기록되어야 함 |
| `batch_id` | 소속 번역 배치 | 동시에 하나의 활성 배치에만 속함 |

## 개체 2: 용어 항목

반복되는 영어 기술 용어와 합의된 한국어 표현을 나타낸다.

| 필드 | 의미 | 검증 규칙 |
|------|------|-----------|
| `source_term` | 원문 용어 | 대소문자 구분 여부를 명시 |
| `korean_term` | 기본 한국어 표현 | 비어 있으면 원형 유지 사유 필요 |
| `keep_original` | 원문 유지 여부 | 식별자·고유명·표준 약어에 사용 |
| `first_use_format` | 최초 사용 시 병기 방식 | 예: `근거 추적성(traceability)` |
| `contexts` | 적용되는 문서 유형 또는 도메인 | 문맥별 의미가 다르면 항목을 분리 |
| `exceptions` | 기본 표현을 적용하지 않는 경우 | 예외 사유 필수 |

## 개체 3: 보호 요소

번역 전후에 정확히 또는 의미상 보존해야 하는 항목을 나타낸다.

| 필드 | 의미 | 검증 규칙 |
|------|------|-----------|
| `kind` | `exact`, `structural`, `semantic` | 세 종류 중 하나 |
| `source_location` | 원문 파일과 위치 | 검토자가 원문을 찾을 수 있어야 함 |
| `original_value` | 원문 값 또는 의미 요약 | 정확 일치 항목은 원문 그대로 기록 |
| `validation_method` | 비교 또는 검토 방법 | 재현 가능한 명령이나 체크 항목 필요 |
| `criticality` | `normal`, `constitution` | constitution 항목은 G4 필수 대상 |

정확 일치 보호 요소에는 다음이 포함된다.

- `FR-*`, `SER-*`, `SC-*`, `T*` 식별자
- 파일 경로, 명령어, URL, 데이터 필드명
- 코드·설정 블록 중 실행 또는 계약을 나타내는 내용
- constitution의 정확한 한국어 면책문

## 개체 4: 번역 배치

한 번에 번역하고 승인할 최소 작업 단위를 나타낸다.

| 필드 | 의미 | 검증 규칙 |
|------|------|-----------|
| `batch_id` | 계획의 P1-A 같은 식별자 | 유일해야 함 |
| `priority` | P1, P2, P3 | 명세 우선순위와 일치 |
| `artifact_paths` | 배치 대상 문서 | 비어 있을 수 없음 |
| `prerequisite_batches` | 선행 승인 배치 | 모두 `approved`여야 시작 가능 |
| `context_paths` | AI가 추가로 읽을 직접 의존 문서 | 최소 필요 범위만 포함 |
| `state` | 배치 처리 상태 | 정의된 상태 전이만 허용 |
| `reviewer_notes` | 의미 결정과 발견 문제 | 실패나 예외가 있으면 필수 |

## 개체 5: 검증 결과

하나의 배치에 대한 게이트 실행 결과를 나타낸다.

| 필드 | 의미 | 검증 규칙 |
|------|------|-----------|
| `batch_id` | 대상 배치 | 존재하는 배치를 참조 |
| `gate` | G0~G5 | 각 필수 게이트당 하나 이상의 결과 필요 |
| `status` | `pass`, `fail`, `blocked` | 실패 사유 없이 `fail` 불가 |
| `checked_at` | 검토 시점 | 기록 형식을 통일 |
| `method` | 사용한 명령 또는 수동 검토 절차 | 재현 가능해야 함 |
| `findings` | 발견 사항 | 실패·차단 시 필수 |
| `evidence` | diff, 명령 출력 또는 체크리스트 참조 | 합격 근거를 찾을 수 있어야 함 |

## 개체 6: 번역 발견 사항

번역 중 발견했지만 같은 변경에서 조용히 고치지 않을 원문 문제를 나타낸다.

| 필드 | 의미 | 검증 규칙 |
|------|------|-----------|
| `finding_id` | 발견 사항 식별자 | 유일해야 함 |
| `path` | 발견 문서 | 실제 문서를 참조 |
| `category` | 모순, 오래된 참조, 누락, constitution 충돌, 오탈자 | 분류 필수 |
| `description` | 문제 설명 | 현재 의미와 예상 영향을 포함 |
| `translation_action` | 번역에서 취한 조치 | 기본값은 의미 보존과 별도 보고 |
| `follow_up` | 후속 명세 또는 이슈 | 필요한 경우 명시 |

## 관계

- 하나의 번역 배치는 하나 이상의 문서 산출물을 포함한다.
- 문서 산출물은 여러 용어 항목과 보호 요소를 참조할 수 있다.
- 번역 배치는 G0~G5 검증 결과를 가진다.
- 문서 산출물은 여러 번역 발견 사항을 가질 수 있다.
- 후속 배치는 선행 배치의 승인 상태와 용어 결정을 재사용한다.

## 상태 전이

### 문서 번역 상태

```text
discovered
  → baselined
  → translating
  → mechanically_verified
  → semantically_reviewed
  → approved
```

- G1 또는 G2 실패: `translating`으로 돌아간다.
- G3 또는 G4 실패: `translating`으로 돌아가 의미를 수정한다.
- G5 실패: 실패 원인에 따라 `translating` 또는 `mechanically_verified`로 돌아간다.
- 범위 제외 문서는 `excluded`로 전환하며 사유가 필수다.

### 배치 상태

```text
planned → ready → in_progress → validating → approved
                                ↘ failed
```

- 선행 배치가 승인되지 않으면 `planned`에서 `ready`로 전환할 수 없다.
- `failed` 배치는 수정 후 `in_progress`로 돌아갈 수 있다.
- 모든 문서와 G0~G5가 통과해야 배치가 `approved`가 된다.

## 전역 검증 규칙

1. 승인된 문서에는 영문 설명을 복제한 주석이나 병렬 번역 섹션이 없어야 한다.
2. 보호 식별자 집합과 링크 목적지는 번역 전후에 동일해야 한다. 의도된 변경은 별도 승인이 필요하다.
3. 숫자, 횟수, 우선순위와 부정 조건은 원문 의미와 같아야 한다.
4. constitution 중요 보호 요소의 실패는 전체 배치를 실패시킨다.
5. 자동 영문 검색 결과만으로 문서를 실패시키지 않고 용어집과 보호 목록을 함께 확인한다.
6. 원문 문제를 수정하는 변경은 번역 배치와 분리한다.
