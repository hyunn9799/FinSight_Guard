# 구현 계획: Spec Kit 문서 한국어화

**브랜치**: `main` | **작성일**: 2026-06-30 | **명세**: [spec.md](./spec.md)

**입력**: `specs/007-korean-spec-docs/spec.md`의 기능 명세와 사용자 요청 "번역 순서와 검증 방법 설계"

## 요약

`specs/` 아래의 영어 설명 문장을 한국어 단일 본문으로 전환하되, 기술 식별자와 설계 의미를 보존한다. 작업은 공통 기준 수립, 핵심 명세, 기능별 후속 산출물, 향후 작성 규칙의 순서로 나누며 각 번역 배치는 기계적 검증과 의미 검토를 모두 통과해야 다음 배치로 진행한다. AI에는 전체 문서가 아니라 대상 배치와 직접 의존 문서만 제공하여 토큰 사용과 문맥 혼선을 제한한다.

## 학습 단계 브리핑

**이번 단계의 목적**: 학습자가 번역 자체보다 요구사항과 설계 결정의 보존 과정을 이해할 수 있도록, 번역 순서와 단계별 품질 게이트를 설계한다.

**이번 단계에서 확인할 판단**:

- 어떤 문서부터 번역해야 후속 문서의 용어와 의미가 안정되는지 결정한다.
- 한국어 설명과 원형을 유지할 기술 요소의 경계를 정한다.
- 자동 검사가 잡을 수 있는 오류와 사람이 검토해야 하는 의미 오류를 분리한다.
- AI가 한 번에 읽을 문서 묶음을 최소화하여 토큰 사용량과 중복 문맥을 줄인다.
- 금융 안전, 근거 추적, 실패 처리와 관측성 요구사항의 의미가 번역 중 약화되지 않도록 별도 검증 게이트를 둔다.

**가정**:

- 번역 원문은 Git 이력에서 복구할 수 있으므로 영문 복제본이나 영문 주석을 추가하지 않는다.
- 번역 실행은 후속 `/speckit-tasks`와 사용자가 선택한 구현 task에서만 수행한다.
- 기계적 영문 탐지는 기술 용어까지 오탐할 수 있으므로 단독 합격 기준으로 사용하지 않는다.
- 현재 문서의 설계 모순이나 오래된 내용은 번역 과정에서 수정하지 않고 별도 이슈로 분리한다.

## 기술 컨텍스트

**언어/버전**: GitHub 호환 Markdown, UTF-8; 검증 명령은 POSIX 계열 셸과 Python 3 표준 도구 사용

**주요 의존성**: Git, `rg`, `find`, `diff`; 새 외부 패키지 없음

**저장소**: Git으로 추적되는 `specs/**/*.md` 파일; 별도 데이터베이스나 런타임 저장소 없음

**테스트**: `git diff --check`, 보호 식별자·숫자·코드 펜스 비교, 상대 링크 확인, 미완성 표시 검색, 의미 보존 체크리스트, 저장소 정책상 `python3 -m compileall src`

**대상 플랫폼**: 로컬 개발 저장소, GitHub Markdown 렌더링, Codex/LLM 문서 컨텍스트

**프로젝트 유형**: 문서 전용 변경

**성능 목표**: 영문과 한국어 중복 본문 0건; AI 검토 컨텍스트는 대상 배치, 용어집, 보호 목록과 직접 의존 문서로 제한; 한 번의 검토 단위는 하나의 기능 또는 하나의 산출물 계층으로 제한

**제약조건**: 런타임 코드 변경 금지, 문서 구조와 식별자 보존, 정확한 면책문 보존, 새 번역 도구 의존성 추가 금지, 배치 실패 시 다음 배치 진행 금지

**규모/범위**: 계획 시작 시점 기준 기능 디렉터리 6개, Markdown 문서 31개, 약 4,859줄. 직접 번역 범위는 `specs/` 아래 Markdown이며 constitution과 일반 프로젝트 문서의 전면 번역은 제외

## Constitution 점검

*게이트: Phase 0 조사 전에 통과해야 하며 Phase 1 설계 뒤 다시 확인한다.*

### 설계 전 점검

- **학습 중심 Spec Kit 거버넌스 — PASS**: 범위, 가정, 대안, 예상 산출물과 검증 이유를 계획 전에 설명했다.
- **근거 기반 금융 리서치 — PASS/보존 대상**: 런타임 리서치 로직은 바꾸지 않으며 `EvidenceItem` 요구사항과 증거 요약 의미를 보호 요소로 검증한다.
- **금융 안전과 시나리오 표현 — PASS/보존 대상**: 정확한 면책문, 직접 권유 금지, 보장 표현 금지와 시나리오 표현을 별도 안전 게이트로 둔다.
- **역할 기반 LangGraph 워크플로 — PASS/비영향**: 에이전트 역할, 조건부 라우팅, 재시도·대체 처리와 최대 2회 재작성 규칙을 변경하지 않는다.
- **결정적 품질 게이트 — PASS**: 외부 API 없이 로컬 파일과 Git 기준선만 사용한다. 자동 검사와 수동 의미 검토의 역할을 분리한다.
- **관측 가능하고 복구 가능한 운영 — PASS/비영향**: 런타임 로그, 보고서 저장, 상태 확인과 지표 동작을 변경하지 않으며 관련 문구의 의미를 보호한다.

**설계 전 Constitution 결과**: PASS. 예외나 위반 없음.

## Phase 0: 조사 결과

상세 결정과 대안은 [research.md](./research.md)에 기록한다. 해결되지 않은 `NEEDS CLARIFICATION` 항목은 없다.

핵심 조사 결론:

1. 영문 주석을 병기하지 않고 한국어 단일 본문을 사용한다.
2. 문서 전체 일괄 처리 대신 의존성과 위험도에 따른 배치를 사용한다.
3. 영문 탐지는 후보 탐색에만 사용하고, 합격은 구조·보호 요소·의미·constitution 게이트로 판단한다.
4. 새 자동 번역 또는 언어 감지 도구를 도입하지 않는다.
5. 원문의 문제는 번역과 분리하여 기록한다.

각 배치는 앞 배치가 승인된 뒤 시작한다. 한 배치에 제공하는 AI 컨텍스트는 다음으로 제한한다.

- 번역 대상 문서
- `glossary.md`의 관련 용어
- 해당 배치의 보호 요소 기준선
- 직접 링크되거나 의미상 선행하는 문서
- constitution의 관련 원칙만 발췌한 검토 기준

### 검증 게이트

1. **G0 기준선 게이트**: 대상 경로, 문서 유형, 상태, 요구사항·작업 ID, 숫자, 링크 목적지, 코드 펜스, 정확 일치 문구를 번역 전에 기록한다.
2. **G1 구조 게이트**: 제목 계층, 표, 목록, 코드 펜스, 요구사항·작업 ID, 링크 목적지와 명령어가 보존되었는지 검사한다.
3. **G2 언어 게이트**: 설명 문장이 한국어인지 검토한다. 영문 검색 결과는 허용 목록과 대조하며 기술 식별자는 실패로 처리하지 않는다.
4. **G3 의미 게이트**: 의무 강도, 부정문, 수치, 우선순위, 범위 경계, 실패 조건과 인수 기준이 원문과 같은지 사람이 문단 단위로 검토한다.
5. **G4 constitution 게이트**: 금융 권유 금지, 보장 표현 금지, 정확한 면책문, `EvidenceItem`, 재시도·대체 처리, 최대 2회 재작성, 관측성 요구를 별도 확인한다.
6. **G5 완료 게이트**: 미완성 표시·깨진 참조·공백 오류가 없고 배치 체크리스트가 모두 통과했는지 확인한다. 실패하면 같은 배치를 수정하고 재검증한다.
