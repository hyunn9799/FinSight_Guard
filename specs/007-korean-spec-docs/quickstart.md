# 빠른 시작: 번역 배치 검증

## 목적

이 가이드는 하나의 번역 배치를 선택하고 구조·식별자·의미·constitution 보호 조건을 검증하는 절차를 설명한다. 전체 번역 구현이나 완성된 테스트 스위트를 제공하지 않는다.

## 사전 조건

- 저장소 루트에서 실행한다.
- 대상 배치와 선행 배치가 `translation-inventory.md`에 기록되어 있다고 가정한다.
- 대상 문서의 원문이 Git `HEAD`에서 조회 가능해야 한다.
- `git`, `rg`, `find`, `diff`, Python 3를 사용할 수 있어야 한다.
- 다른 사용자의 작업 트리 변경을 덮어쓰지 않는다.

## 1. 배치 범위 확인

예시로 하나의 기존 명세를 선택한다.

```bash
TARGET=specs/001-robust-backtest-validation/spec.md
git status --short -- "$TARGET"
```

기대 결과:

- 대상 경로가 계획된 배치와 일치한다.
- 번역 전의 예상하지 못한 변경이 있으면 진행을 중단하고 소유자를 확인한다.

전체 문서 현황은 다음 명령으로 확인한다.

```bash
find specs -type f -name '*.md' -print | sort
```

## 2. 보호 식별자 기준선 비교

번역 전 Git 기준선과 현재 문서의 요구사항·작업 식별자를 비교한다.

```bash
git show "HEAD:$TARGET" | rg -o '(FR|SER|SC)-[0-9]{3}|T[0-9]{3}' | sort -u > /tmp/finsight-before-ids.txt
rg -o '(FR|SER|SC)-[0-9]{3}|T[0-9]{3}' "$TARGET" | sort -u > /tmp/finsight-after-ids.txt
diff -u /tmp/finsight-before-ids.txt /tmp/finsight-after-ids.txt
```

기대 결과: 출력이 없고 종료 코드가 0이다. 차이가 있으면 의도된 요구사항 변경인지 확인하며, 번역 task에서는 변경을 허용하지 않는다.

같은 방식으로 숫자와 코드 펜스 개수를 후보 검사한다.

```bash
git show "HEAD:$TARGET" | rg -o '[0-9]+([.][0-9]+)?%?' | sort > /tmp/finsight-before-numbers.txt
rg -o '[0-9]+([.][0-9]+)?%?' "$TARGET" | sort > /tmp/finsight-after-numbers.txt
diff -u /tmp/finsight-before-numbers.txt /tmp/finsight-after-numbers.txt
git show "HEAD:$TARGET" | rg -c '^```'
rg -c '^```' "$TARGET"
```

숫자 차이는 자동 실패 후보이며 문맥을 확인한다. 코드 펜스 수는 같아야 하지만, 같다는 사실만으로 내용 보존이 증명되지는 않는다.

## 3. 구조와 미완성 표시 검사

```bash
git diff --check -- "$TARGET"
rg -n 'NEEDS CLARIFICATION|ACTION REQUIRED|TODO|TBD|\[FEATURE NAME\]|\[DATE\]' "$TARGET"
rg -n '^#{1,6} |^```|\]\(' "$TARGET"
```

기대 결과:

- `git diff --check`가 성공한다.
- 의도하지 않은 자리표시자나 미완성 표시가 없다.
- 제목 계층, 코드 펜스와 링크가 원문 구조에 대응한다.

상대 링크는 각 목적지가 실제로 존재하는지 확인한다. 링크 표시 문장은 번역할 수 있지만 괄호 안 목적지는 변경하지 않는다.

## 4. 언어와 용어 검사

영문이 포함된 줄을 후보로 찾는다.

```bash
rg -n '[A-Za-z]{4,}' "$TARGET"
```

각 결과를 다음 중 하나로 분류한다.

- 번역해야 하는 설명 문장
- 원형을 유지하는 코드·필드·경로·명령어·URL
- 용어집이 허용한 제품명·약어·최초 병기
- 정확성을 위해 보존한 코드 블록

기대 결과: 번역되지 않은 설명 문장은 0건이다. 영어 검색 결과 자체가 0일 필요는 없다.

## 5. 의미 보존 검토

`git diff --word-diff`로 원문과 번역문을 문단 단위로 검토한다.

```bash
git diff --word-diff -- "$TARGET"
```

다음을 확인한다.

- 의무와 권고의 강도가 달라지지 않았다.
- 부정문이나 제외 범위가 긍정 또는 선택 사항으로 바뀌지 않았다.
- 숫자, 우선순위, 재시도 횟수, 기간과 종료 조건이 같다.
- 사용자 스토리, 인수 시나리오와 성공 기준이 누락되지 않았다.
- 문서 소유권과 기능 간 경계가 바뀌지 않았다.
- 새 설계 결정이 번역문에 추가되지 않았다.

## 6. Constitution 중요 항목 검사

대상 문서에 관련 요구사항이 있다면 다음을 확인한다.

- 직접적인 매수·매도·보유 권유와 실거래 실행을 허용하지 않는다.
- 수익, 목표가 또는 원금 보장 표현을 허용하지 않는다.
- `EvidenceItem`과 근거 요약 요구가 보존된다.
- 데이터 누락과 provider 실패를 공개하고 대체 동작을 유지한다.
- 시장 데이터 실패 재시도 1회와 평가 실패 후 최대 2회 재작성 조건이 바뀌지 않는다.
- 구조화 로그, 보고서 저장, 상태 확인과 지표 요구가 보존된다.
- 최종 보고서 면책문이 필요한 문서에서는 다음 문장이 정확히 유지된다.

```text
본 보고서는 교육 및 정보 제공 목적의 AI 리서치 결과이며, 특정 종목의 매수·매도·보유를 권유하지 않습니다. 최종 투자 판단과 책임은 투자자 본인에게 있습니다.
```

금지 표현 검색 결과는 문맥을 검토한다. 금지 규칙을 설명하기 위해 등장한 경우와 실제 권유 문장을 구분해야 한다.

## 7. 배치 완료 검사

```bash
git diff --check
python3 -m compileall src
git status --short
```

문서 전용 배치이므로 런타임 pytest는 기본적으로 필요하지 않다. 제품 코드나 테스트 파일이 함께 바뀌었다면 번역 범위를 위반한 것이므로 변경 원인을 확인하고 관련 테스트를 별도 수행한다.

배치는 다음 조건을 모두 만족할 때만 승인한다.

- G0~G5 체크리스트 통과
- 미번역 설명 문장 0건
- 승인되지 않은 보호 요소 변경 0건
- 깨진 상대 링크 0건
- constitution 중요 의미 누락·완화 0건
- 원문 문제는 별도 발견 사항으로 기록

## 실패 처리

- 기계 검증 실패는 해당 문서를 수정한 뒤 G1부터 다시 실행한다.
- 의미 또는 constitution 검증 실패는 번역 단계로 돌아가 G3 또는 G4를 다시 실행한다.
- 선행 배치가 승인되지 않았거나 원문 충돌이 발견되면 현재 배치를 차단한다.
- 실패한 배치를 건너뛰고 다음 배치를 시작하지 않는다.
