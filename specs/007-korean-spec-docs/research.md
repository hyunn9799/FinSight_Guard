# 조사: Spec Kit 문서 한국어화 전략

## 조사 목적과 범위

이 조사는 외부 번역 서비스나 새로운 도구를 선정하기 위한 것이 아니다. 저장소 안의 문서 구조, 기능 간 의존성, constitution 제약과 AI 컨텍스트 비용을 기준으로 번역 순서와 검증 방식을 결정한다.

계획 시작 시점의 로컬 조사 결과는 다음과 같다.

- `specs/` 아래 기능 디렉터리: 6개
- Markdown 문서: 31개
- 전체 분량: 약 4,859줄
- 후속 설계 산출물이 있는 기능: `002`, `004`, `006`
- 명세만 있거나 핵심 문서가 적은 기능: `001`, `003`
- 이미 한국어 기준으로 작성된 기능: `007`

조사 명령은 `find`, `wc`, `rg`를 사용했으며 외부 네트워크나 실시간 API를 사용하지 않았다.

## 결정 1: 한국어 단일 본문

**결정**: 설명 문장은 한국어로 대체하고 영문 번역본을 주석, 병렬 섹션 또는 별도 복제 파일로 남기지 않는다. 코드 식별자, 필드명, 경로, 명령어, URL, 표준 약어와 고유 제품명은 원형을 유지한다.

**근거**:

- Markdown 주석도 AI 컨텍스트에 포함되어 토큰을 소비한다.
- 영한 병기는 같은 의미를 두 번 제공해 문서 길이와 검토 비용을 늘린다.
- 두 언어가 서로 다르게 갱신되는 드리프트를 막기 어렵다.
- 원문은 Git 이력에서 복구할 수 있다.

**검토한 대안**:

- 영문 아래 한국어 주석: 읽기와 렌더링은 편하지만 토큰과 동기화 비용이 가장 크다.
- 영문·한국어 파일 분리: 각 언어의 가독성은 좋지만 파일 수와 참조 경로가 두 배가 된다.
- 영어만 유지: 토큰 수는 작을 수 있지만 학습 목적을 충족하지 못한다.

## 결정 2: 기능 의존성을 따른 두 번의 통과

**결정**: 첫 번째 통과에서 모든 `spec.md`와 요구사항 체크리스트를 기능 의존성 순서로 번역하고, 두 번째 통과에서 후속 산출물이 있는 기능을 하나씩 완성한다.

**근거**:

- 명세의 용어와 범위가 후속 계획·작업 문서의 기준이다.
- `001`의 견고성 검증 개념은 `002`의 walk-forward 최적화에 선행한다.
- `003`의 저장소 방향은 `004`의 PostgreSQL 기준 스키마로 구체화된다.
- `004`의 저장 경계는 `006`의 provider 정규화·영속성 계약에 선행한다.
- 한 기능 묶음을 완성하면 문서 간 링크와 요구사항 추적성을 같은 검토에서 확인할 수 있다.

**검토한 대안**:

- 파일명 종류별 전체 번역: 같은 종류를 빠르게 처리할 수 있지만 기능별 문맥 전환이 잦다.
- 디렉터리 번호만 따른 전체 번역: 단순하지만 명세 용어를 먼저 고정하지 못한다.
- 모든 문서 동시 번역: 가장 빠르게 보이지만 검토 범위가 커져 누락과 의미 드리프트의 원인을 찾기 어렵다.

## 결정 3: 최소 AI 컨텍스트 배치

**결정**: AI는 한 번에 전체 문서 집합을 읽지 않는다. 대상 문서, 관련 용어집 항목, 보호 요소 기준선, 직접 의존 문서와 관련 constitution 원칙만 컨텍스트로 사용한다.

**근거**:

- 전체 문서를 반복 제공하면 토큰 사용량이 문서 수에 비례해 커진다.
- 관련 없는 도메인 용어가 섞이면 같은 영어 단어를 서로 다르게 번역할 가능성이 높아진다.
- 작은 배치는 diff와 의미 검토의 책임 범위를 명확하게 만든다.

**검토한 대안**:

- 전체 `specs/`를 매번 제공: 전역 일관성은 보이지만 비용과 문맥 혼선이 크다.
- 대상 문서만 제공: 비용은 최소지만 선행 용어와 교차 문서 계약을 놓칠 수 있다.
- 문서 요약만 제공: 비용은 줄지만 수치, 부정문, 정확한 조건을 잃을 수 있다.

## 결정 4: 보호 요소 분류

**결정**: 번역 전에 각 배치에서 보호 요소를 다음 세 종류로 분류한다.

1. **정확 일치 요소**: 요구사항·작업 ID, 경로, 명령어, URL, 데이터 필드명, 정확한 면책문.
2. **구조 보존 요소**: 제목 계층, 목록 순서, 표 열, 링크 목적지, 코드 펜스, 문서 간 참조.
3. **의미 보존 요소**: MUST/SHOULD에 해당하는 의무 강도, 부정문, 수치·횟수, 우선순위, 실패·대체 조건, 범위 제외.

**근거**: 모든 내용을 같은 방식으로 비교하면 번역 가능한 문장과 변경하면 안 되는 식별자를 구분할 수 없다.

**검토한 대안**:

- 전체 파일 바이트 비교: 번역 자체를 허용할 수 없다.
- 코드 블록만 보호: 요구사항 ID, 링크와 면책문을 놓친다.
- 사람의 의미 검토만 수행: 반복적인 구조 오류를 비효율적으로 확인하게 된다.

## 결정 5: 다층 검증

**결정**: 각 배치는 기준선, 구조, 언어, 의미, constitution, 완료의 여섯 게이트를 순서대로 통과해야 한다. 하나라도 실패하면 다음 배치를 시작하지 않는다.

**근거**:

- 기계 검사는 ID·링크·숫자·형식 누락에 강하지만 자연스러운 번역과 의무 강도는 판단하지 못한다.
- 사람 검토는 의미를 판단할 수 있지만 반복되는 식별자 비교에서 실수할 수 있다.
- 금융 안전과 근거 요구사항은 일반 의미 검토에 묻히지 않도록 별도 확인해야 한다.

**검토한 대안**:

- 영문 문자열이 없으면 합격: 기술 용어 때문에 오탐이 많고 의미 손실은 탐지하지 못한다.
- 리뷰어 승인만 사용: 일관된 재현과 누락 추적이 어렵다.
- 자동 번역 품질 점수 사용: 새 외부 의존성과 불투명한 합격 기준이 생긴다.

## 결정 6: 원문 문제와 번역 변경 분리

**결정**: 원문에서 모순, 오래된 참조, 잘못된 수치 또는 constitution 충돌을 발견하면 번역 diff 안에서 조용히 고치지 않는다. 번역은 현재 의미를 보존하고 문제를 별도 검토 항목으로 기록한다.

**근거**: 번역과 설계 변경을 한 diff에 섞으면 어떤 변경이 언어 변환이고 어떤 변경이 요구사항 수정인지 추적하기 어렵다.

**검토한 대안**:

- 발견 즉시 수정: 문서는 빨리 깨끗해지지만 승인되지 않은 범위 변경이 된다.
- 문제를 무시: 번역은 끝나도 학습자가 잘못된 설계를 신뢰할 수 있다.

## 결정 7: 새 도구를 추가하지 않음

**결정**: 기존 Git, `rg`, `find`, `diff`, Python 3 표준 환경만 사용한다. 전용 번역 라이브러리, 언어 감지 모델, Markdown 검사 패키지는 이번 기능에 추가하지 않는다.

**근거**:

- 현재 규모는 기능 단위 배치와 체크리스트로 관리 가능하다.
- 새 도구의 설치·버전·오탐 관리가 번역 자체보다 큰 인프라가 될 수 있다.
- 결정적이고 오프라인인 검증을 유지할 수 있다.

**검토한 대안**:

- 자동 번역 API: 속도는 높지만 외부 의존성, 비용, 비결정성과 민감 문구 훼손 위험이 생긴다.
- Markdown 링크 검사기 추가: 편리하지만 현재 범위에서는 로컬 상대 링크 검토로 충분하다.
- 언어 감지 라이브러리 추가: 기술 식별자가 많은 문서에서 신뢰하기 어렵다.

## 조사 결론

미해결 항목은 없다. 한국어 단일 본문, 의존성 기반 배치, 최소 AI 컨텍스트, 보호 요소 분류와 여섯 단계 검증 게이트를 계획에 적용한다.
