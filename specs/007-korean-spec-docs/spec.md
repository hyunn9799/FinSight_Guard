# 기능 명세: Spec Kit 문서 한국어화

**기능 브랜치**: `[007-korean-spec-docs]`

**작성일**: 2026-06-30

**상태**: 초안

**입력**: 사용자 설명: "constitution 문서를 먼저 확인하고, MVP 이상의 결과물을 목표로 하면서 학습용으로 개발 중이므로 영어로 된 Spec Kit 문서들을 모두 한국어로 바꾼다."

## 학습 단계 브리핑

**이번 단계의 목적**: 영어 독해 부담 없이 요구사항에서 계획과 작업으로 이어지는 설계 결정을 학습할 수 있도록, Spec Kit 산출물의 한국어화 범위와 의미 보존 기준을 정의한다.

**이번 단계에서 확인할 판단**:

- 기존 문서와 앞으로 생성될 문서 중 한국어화할 범위를 정한다.
- 번역해야 하는 설명 문장과 원형을 유지해야 하는 기술 식별자의 경계를 정한다.
- 금융 안전, 근거 추적, 실패 처리, 관측성 요구사항이 번역 과정에서 약화되지 않도록 검토 기준을 정한다.
- 한 번에 모든 문서를 바꾸지 않고 학습과 검토가 가능한 우선순위로 나눈다.

**가정**:

- "spec 문서들"은 `specs/` 아래의 모든 Markdown 산출물을 뜻하는 것으로 해석한다.
- constitution은 이번 단계의 상위 원칙으로 검토하지만, constitution 자체의 번역은 이 기능의 직접 범위에 포함하지 않는다.
- 코드 식별자, 데이터 필드명, 파일 경로, 명령어, 고유 제품명과 널리 쓰이는 기술 약어는 정확성을 위해 원문을 유지할 수 있다.
- 기존 산출물의 설계 결정은 번역으로 변경하지 않으며, 내용 변경이 필요하면 별도의 명세 변경으로 다룬다.

## 사용자 시나리오 및 테스트 *(필수)*

### 사용자 스토리 1 - 핵심 명세를 한국어로 학습 (우선순위: P1)

학습자는 각 기능의 `spec.md`와 요구사항 품질 체크리스트를 한국어로 읽고, 사용자 가치, 범위, 요구사항, 성공 기준과 가정을 영어 문장 해석 없이 파악한다.

**이 우선순위인 이유**: 명세는 이후 계획과 작업 목록의 기준이므로, 이 문서부터 한국어화해야 잘못된 요구사항 해석이 후속 단계로 전파되지 않는다.

**독립 테스트**: 기존 기능 하나의 `spec.md`와 체크리스트만 한국어화한 뒤, 원문 설계 결정과 요구사항 식별자가 모두 보존되고 한국어만으로 기능 범위를 설명할 수 있는지 검토한다.

**인수 시나리오**:

1. **전제** 영어 설명이 포함된 기존 기능 명세가 있을 때, **행동** 학습자가 한국어화된 명세를 읽으면, **결과** 사용자 스토리, 요구사항, 성공 기준, 가정을 추가 번역 도구 없이 설명할 수 있다.
2. **전제** 명세에 요구사항 ID, 파일 경로, 명령어와 데이터 필드명이 있을 때, **행동** 문서를 한국어화하면, **결과** 해당 기술 식별자는 원문과 동일하게 유지되고 설명 문장만 자연스러운 한국어가 된다.
3. **전제** 금융 안전과 근거 추적 요구사항이 있을 때, **행동** 번역 검토를 수행하면, **결과** 금지 범위, 정확한 면책문과 검증 조건이 누락되거나 완화되지 않는다.

---

### 사용자 스토리 2 - 설계 과정 전체를 한국어로 추적 (우선순위: P2)

학습자는 계획, 조사, 데이터 모델, 계약, 빠른 시작, 작업 목록을 한국어로 따라가며 하나의 요구사항이 설계 결정과 구현 작업으로 구체화되는 과정을 추적한다.

**이 우선순위인 이유**: 핵심 명세만 한국어여도 최소 학습 가치는 생기지만, Spec Kit의 단계 간 추적성을 공부하려면 후속 산출물도 같은 언어와 용어 체계를 사용해야 한다.

**독립 테스트**: 한 기능 디렉터리의 모든 Markdown 산출물을 한국어화한 뒤, 요구사항 ID와 문서 간 링크를 따라 명세에서 계획 및 작업까지 동일한 결정이 유지되는지 검토한다.

**인수 시나리오**:

1. **전제** 하나의 기능에 여러 단계의 영어 산출물이 있을 때, **행동** 해당 기능의 문서 묶음을 한국어화하면, **결과** 학습자는 문서 간 링크와 요구사항 ID를 따라 설계 흐름을 끊김 없이 추적할 수 있다.
2. **전제** 서로 다른 문서에서 같은 개념을 표현할 때, **행동** 번역 결과를 비교하면, **결과** 공통 용어집에 정의된 한국어 표현과 원문 기술 용어가 일관되게 사용된다.
3. **전제** 아직 번역되지 않은 문서가 남아 있을 때, **행동** 학습자가 문서 현황을 확인하면, **결과** 완료·검토 필요·제외 상태와 사유를 구분할 수 있다.

### 사용자 스토리 3 - 향후 산출물을 처음부터 한국어로 작성 (우선순위: P3)

기여자는 새로운 Spec Kit 산출물을 처음부터 자연스러운 한국어로 작성하고, 필요한 기술 용어만 원문을 병기하여 기존 문서를 반복 번역하지 않는다.

**이 우선순위인 이유**: 기존 문서 번역 뒤에도 영어 템플릿을 계속 사용하면 혼합 언어 상태가 재발하므로, 장기적인 학습 환경을 위해 작성 규칙이 필요하다.

**독립 테스트**: 새 기능 명세를 생성하여 제목, 설명, 시나리오, 요구사항과 체크리스트가 한국어로 작성되고 필수 구조 및 품질 기준은 유지되는지 검토한다.

**인수 시나리오**:

1. **전제** 새 Spec Kit 산출물을 작성할 때, **행동** 기여자가 문서 작성 규칙을 적용하면, **결과** 모든 설명 문장은 한국어이고 기술 식별자만 필요한 범위에서 원문으로 유지된다.
2. **전제** 한국어로 옮기면 의미가 모호해지는 기술 용어가 있을 때, **행동** 해당 용어를 처음 사용하면, **결과** 합의된 한국어 표현과 원문 용어를 함께 제시하고 이후 동일한 표현을 사용한다.

### 경계 사례

- 한국어로 번역하면 코드 식별자 또는 외부 계약의 정확한 필드명이 달라지는 경우
- 한 문서에서 같은 영어 용어가 문맥에 따라 서로 다른 의미로 사용되는 경우
- 정확히 일치해야 하는 면책문, 명령어, 경로, URL 또는 스키마명이 포함된 경우
- 원문 문서 자체에 모순, 오래된 링크, 미완성 자리표시자 또는 오탈자가 있는 경우
- 영어와 한국어가 섞인 문서에서 이미 자연스럽게 정착된 기술 용어까지 과도하게 번역할 위험이 있는 경우
- 이전 기능의 문서가 현재 constitution 또는 후속 명세와 충돌하는 경우
- 번역 중 문서 간 링크, 요구사항 ID 또는 작업 ID가 변경되는 경우

## 요구사항 *(필수)*

### 기능 요구사항

- **FR-001**: 문서 한국어화 범위는 `specs/` 아래의 모든 Markdown 산출물을 식별하고, 각 문서의 번역 상태를 확인할 수 있어야 한다.
- **FR-002**: 범위에 포함된 문서의 제목, 설명, 근거, 시나리오, 요구사항, 성공 기준, 가정과 검토 메모는 자연스러운 한국어로 제공되어야 한다.
- **FR-003**: 코드 식별자, 요구사항 및 작업 ID, 데이터 필드명, 파일 경로, 명령어, URL, 고유 제품명과 표준 기술 약어는 참조 정확성을 위해 원형을 유지해야 한다.
- **FR-004**: 한국어화된 문서는 원문의 섹션 순서, 우선순위, 요구사항 의미, 범위 경계, 인수 조건과 성공 기준을 보존해야 한다.
- **FR-005**: 문서 간 상대 링크와 요구사항·작업 추적 관계는 번역 전후에 동일하게 유효해야 한다.
- **FR-006**: 반복되는 핵심 용어는 하나의 합의된 한국어 표현을 사용해야 하며, 첫 사용 시 원문 병기가 필요한 용어를 구분해야 한다.
- **FR-007**: 번역 과정에서 발견한 원문의 모순, 누락, 오래된 정보 또는 constitution 충돌은 번역으로 조용히 수정하지 않고 별도의 검토 항목으로 기록해야 한다.
- **FR-008**: 문서 한국어화는 기능 디렉터리 또는 산출물 단계 단위로 검토 가능하게 수행되어야 하며, 각 단위는 변경 목적, 핵심 번역 결정, 검토한 대안과 확인 결과를 남겨야 한다.
- **FR-009**: P1 범위는 모든 `spec.md`와 `checklists/requirements.md`를 포함해야 한다.
- **FR-010**: P2 범위는 `plan.md`, `tasks.md`, `research.md`, `data-model.md`, `quickstart.md`, `contracts/*.md`와 그 밖의 `specs/` 하위 Markdown 산출물을 포함해야 한다.
- **FR-011**: P3 범위는 향후 생성되는 Spec Kit 산출물이 처음부터 한국어 설명을 사용하도록 문서 작성 기준과 품질 검토 기준을 제공해야 한다.
- **FR-012**: 문서 한국어화 작업은 제품 실행 코드, 데이터 저장 내용 또는 외부 서비스 동작을 변경해서는 안 된다.
- **FR-013**: 각 번역 단위는 원문 대비 누락, 의미 변경, 미번역 설명 문장과 깨진 참조를 확인하는 품질 검토를 통과해야 한다.

### 안전성, 근거 및 신뢰성 요구사항

- **SER-001**: 번역은 직접적인 매수·매도·보유 권유, 거래 지시, 수익 또는 목표가 보장, 원금 보장, 실거래 실행을 금지하는 기존 원칙을 약화해서는 안 된다.
- **SER-002**: 중요한 수치와 사실을 `EvidenceItem`으로 뒷받침해야 한다는 기존 요구사항과 필드 의미를 그대로 보존해야 한다.
- **SER-003**: constitution에 정의된 다음 면책문은 글자와 문장부호를 포함해 정확히 유지해야 한다.

  ```text
  본 보고서는 교육 및 정보 제공 목적의 AI 리서치 결과이며, 특정 종목의 매수·매도·보유를 권유하지 않습니다. 최종 투자 판단과 책임은 투자자 본인에게 있습니다.
  ```

- **SER-004**: 데이터 누락, 공급자 실패, 성능 저하 상태와 한계를 공개해야 한다는 기존 요구사항을 번역 중 생략하거나 정상 동작으로 오해하게 표현해서는 안 된다.
- **SER-005**: 검증 실패, 공급자 실패, 평가 실패, 재작성 제한과 같은 조건부 흐름은 원문의 횟수, 종료 조건과 대체 동작을 보존해야 한다.
- **SER-006**: 구조화 로그, 보고서 저장, 상태 확인과 지표에 관한 기존 관측성 요구사항은 의미와 측정 항목을 보존해야 한다.

### 핵심 개체

- **문서 산출물**: `specs/` 아래의 Markdown 파일로, 기능 식별자, 산출물 단계, 원문 언어, 번역 상태와 검토 상태를 가진다.
- **용어 항목**: 원문 용어, 합의된 한국어 표현, 원문 병기 여부, 사용 맥락과 예외를 나타낸다.
- **번역 검토 기록**: 대상 문서, 검토한 요구사항 ID, 의미 보존 결과, 발견한 문제와 후속 조치를 나타낸다.
- **범위 목록**: 번역 대상 문서와 완료·검토 필요·제외 상태 및 제외 사유를 한눈에 확인하기 위한 기준 목록이다.

## 성공 기준 *(필수)*

### 측정 가능한 결과

- **SC-001**: P1 완료 시 기존 모든 `spec.md`와 요구사항 품질 체크리스트에서 번역 대상 설명 문장의 100%가 한국어로 제공된다.
- **SC-002**: P2 완료 시 `specs/` 아래에서 번역 대상으로 분류된 모든 Markdown 문서의 설명 문장 100%가 한국어로 제공된다.
- **SC-003**: 검토 표본의 요구사항 ID, 작업 ID, 명령어, 경로, 데이터 필드명과 문서 간 링크는 번역 전후 100% 동일하거나 유효하다.
- **SC-004**: 금융 안전, 근거 추적, 실패 처리와 관측성 관련 필수 요구사항은 원문 대비 누락 또는 의미 완화가 0건이다.
- **SC-005**: 각 번역 단위는 미번역 설명 문장, 의미가 달라진 요구사항, 깨진 문서 참조가 0건인 품질 검토를 통과한다.
- **SC-006**: 한국어화된 기능 문서 하나를 처음 읽는 검토자는 별도의 영어 번역 도움 없이 사용자 가치, 범위, 우선순위와 성공 기준을 모두 식별할 수 있다.
- **SC-007**: 새로 작성되는 Spec Kit 산출물의 필수 설명 섹션은 최초 초안부터 100% 한국어로 작성된다.

## 가정

- 사용자 요청의 "영어로 된 문서들"은 저장소 전체 문서가 아니라 `specs/` 아래의 Spec Kit 산출물을 의미한다.
- constitution, `AGENTS.md`, `PROJECT_PLAN.md`, 템플릿과 일반 프로젝트 문서의 전면 한국어화는 별도 거버넌스 변경 또는 후속 범위로 다룬다.
- 기존 영어 문서의 원문은 버전 관리 이력에서 확인할 수 있으므로 별도의 영문 복제본을 유지하지 않는다.
- 널리 쓰이는 기술 용어를 억지로 번역하기보다 학습자가 원문 자료와 연결할 수 있도록 한국어 설명과 원문 용어를 함께 사용할 수 있다.
- 현재 기능들의 설계 내용은 승인된 것으로 간주하며, 이번 기능은 내용 재설계가 아니라 언어와 학습 접근성 개선에 집중한다.
- 번역은 P1, P2, P3 순서로 검토하며, 각 단계는 독립적으로 완료 상태를 판단할 수 있다.

## 단계 완료 리뷰

- **이번 단계의 목적**: Spec Kit 산출물 전체의 한국어화 목표를 검토 가능한 범위와 성공 기준으로 바꾸는 것이다.
- **새로 생긴 파일 또는 변경된 파일**: `specs/007-korean-spec-docs/spec.md`, `specs/007-korean-spec-docs/checklists/requirements.md`, `.specify/feature.json`.
- **핵심 설계 결정**: 모든 `specs/**/*.md`를 최종 범위로 두되, 핵심 명세·체크리스트, 후속 설계 산출물, 향후 작성 규칙의 세 단계로 나눈다. 기술 식별자는 원형을 유지하고 설명 문장만 한국어화한다.
- **검토한 대안**: 모든 문서를 한 번에 번역하는 방식은 변경량이 커서 의미 손실을 검토하기 어렵다. `spec.md`만 번역하는 방식은 후속 설계 흐름 학습에 부족하다.
- **현재 설계를 선택한 이유**: 단계별로 독립적인 학습 가치를 제공하면서도 최종적으로 모든 Spec Kit 산출물을 일관된 한국어 문서 체계로 만들 수 있기 때문이다.
- **반드시 읽어야 할 부분**: `사용자 스토리 1~3`의 단계 구분, `FR-003~FR-008`의 의미 보존 기준, `SER-001~SER-006`의 constitution 보호 조건, `가정`의 직접 범위와 제외 범위.
